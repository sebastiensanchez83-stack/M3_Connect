13. Gaps & Decisions Needed

GAP 1: Admin persona excluded from frontend admin panel
AuthContext checks persona === 'moderator' only. Users with persona='admin' can't access the admin panel in the frontend, even though the DB is_moderator() function handles both.
Recommendation: Update AuthContext to check both admin and moderator.
GAP 2: 'individual' persona not implemented
Exists in the DB enum but has no onboarding form, no specific profile table, and no feature access defined. Currently a user cannot select it during onboarding.
Recommendation: Either implement it or remove it from the enum.
GAP 3: Moderators can't preview marina-only content
The access check persona === 'marina' && isVerified excludes moderators. They can create/edit marina-only content in the admin panel but can't see it on the public-facing pages.
Recommendation: Add || isModerator to the marina access check.
GAP 4: Pending users can access member-level content
The "members" access level check only verifies !!user (logged in), not isVerified. A user who just signed up and hasn't been approved yet can view member-only resources and events.
Recommendation: Require verified status for member content.
GAP 5: Tier-based restrictions are display-only
The pricing page shows what each tier includes, but no code enforces these restrictions. All verified users get the same access regardless of tier. Features like "Submit bids" or "Priority support" are not gated.
Recommendation: Implement tier checks when the payment system is ready.
GAP 6: No route-level protection
All routes are technically accessible via URL. Protection happens at the component level, meaning a user might see a brief flash of a page before being redirected. Routes like /admin are not wrapped in a guard.
Recommendation: Add a ProtectedRoute wrapper component.
GAP 7: No email notifications
When admin approves/rejects a user, there is no email notification. The user only discovers their status change when they log back in and check their account page.
Recommendation: Add Supabase Edge Functions for email notifications on status changes.
GAP 8: DB tables created but not yet implemented in frontend
These tables exist in Supabase but have no frontend UI yet:
  • subscriptions — Stripe payment (not started)
  • entitlements — Feature access control (not started)
  • partner_requests — B2B partner↔marina contact (not started)
  • rfps — Marina RFP submissions (not started)
  • consultations — Marina consultation requests (not started)
  • content_items, content_drafts, content_approvals — CMS (not started)
  • resource_drafts, resource_approvals — Resource workflow (not started)
